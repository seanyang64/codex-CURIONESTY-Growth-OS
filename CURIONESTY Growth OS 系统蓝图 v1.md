# CURIONESTY Growth OS 好奇成长系统蓝图 v1

版本：v1  
日期：2026-06-04  
定位：给 CURIONESTY 好奇学习社区及未来多项目使用的学生成长管理系统蓝图

---

## 1. 一句话定义

CURIONESTY Growth OS 好奇成长系统，是一个围绕学生长期成长建立的教育管理系统。

它不是传统教务系统，而是一个以“学生唯一档案”为中心，把动力中心、游戏中心、通识中心，以及自治、阅读、通识、行动中的真实记录，持续归纳为成长档案、成绩单、项目履历和教育判断的系统。

核心目标：

> 让每个学生在不同课程、项目、老师、平台和时间中的成长痕迹，都能被看见、被整理、被查询、被复盘、被生成档案。

## 2. 系统设计原则

### 2.1 一生一档案

每个学生拥有一个唯一学号。  
无论学生参加 KIID SCHOOL、好奇学习社区、好奇新生社区，还是好奇·清迈，所有记录都归到同一个学生主档案下。

不能依赖姓名、昵称、英文名来匹配学生，因为会出现重名、改名、拼写不一致、历史资料散落等问题。

### 2.2 先有数据底座，再做功能界面

系统必须先定义稳定的数据结构：

- 学生是谁
- 谁记录了什么
- 在什么时间和场景中发生
- 属于哪个中心
- 属于哪条成长路径
- 是否关联课程、项目、游戏、作品、缴费或成绩

界面可以逐步改，但底层数据结构不能频繁推倒重来。

### 2.3 老师记录必须足够轻

老师端第一优先级是“方便记录”。  
尤其是动力中心，需要允许老师在任意时刻快速记录当下看到的学生状态。

理想状态：

- 30 秒内完成一条记录
- 支持手机端
- 支持标签选择
- 支持语音转文字
- 支持事后补充
- 自动归入学生成长档案

### 2.4 所有记录都要能归档、查询、生成报告

任何一条记录都不只是“写完就算了”。  
它应该能进入：

- 学生成长时间线
- 阶段性成长报告
- 老师工作记录
- 家长沟通材料
- 学生作品集
- 项目复盘
- 管理后台查询

### 2.5 从第一天预留外部平台导入口

CURIONESTY 过去已有大量资料分布在：

- QUIP
- Google Classroom
- Notion
- Google Sheet
- Google Drive
- Gmail
- Excel
- PDF
- 老师个人文档

系统第一版不一定全部自动同步，但必须设计“数据导入与归档中心”，避免未来资料继续分裂。

## 5.2 动力中心

动力中心是 Growth OS 的核心观察系统。

它允许每位老师在任意教育场景中，用“好奇、勇气、觉悟”三个维度记录学生当下被看见的事实，并给出一个具体行动建议。这些即时记录会自动归纳进学生成长档案，形成可追踪、可复盘、可生成报告的成长证据。

### 动力中心记录字段

每条记录包含：

- 学生
- 记录老师
- 记录时间
- 发生时间
- 场景
- 中心：动力中心
- 维度：好奇 / 勇气 / 觉悟
- 事实标签
- 事实描述
- 状态倾向：正向出现 / 需要支持 / 明显波动
- 行动建议
- 是否需要导师跟进
- 是否适合进入家长报告
- 是否适合给学生本人看
- 附件：图片、文字、链接、语音转写

### 三个维度

好奇：

- 主动提问
- 深入追问
- 探索未知
- 发现问题
- 阅读延伸
- 连接经验
- 对他人观点感兴趣

勇气：

- 公开表达
- 主动尝试
- 面对失败
- 进入陌生场景
- 承担挑战
- 发起行动
- 坚持完成

觉悟：

- 自我反思
- 情绪觉察
- 关系觉察
- 承担责任
- 修正行为
- 理解规则
- 看见他人
- 形成判断

### 行动建议

行动建议是动力记录的出口。  
每条动力记录都应该引导下一步支持。

常见建议：

- 继续鼓励提问
- 给一次公开表达机会
- 安排一次一对一谈话
- 邀请担任小组角色
- 给更明确的任务边界
- 帮助复盘冲突
- 推荐一本书或材料
- 安排一次游戏复盘发言
- 暂时降低压力，先恢复安全感
- 提醒导师重点关注

### 动力中心输出

- 学生动力时间线
- 好奇/勇气/觉悟成长画像
- 导师跟进列表
- 阶段性动力报告
- 家长沟通摘要
- 学生自我复盘材料

游戏中心用于记录学生在游戏中的学习、关系、策略和情绪表现。

游戏不是娱乐附属，而是观察学生的重要场景。

记录字段：

- 游戏名称
- 游戏类型
- 参与学生
- 带领老师
- 时间
- 场景
- 规则理解
- 策略思考
- 合作沟通
- 领导与跟随
- 情绪表现
- 面对输赢
- 复盘表达
- 个人贡献
- 团队表现
- 老师观察
- 学生自评
- 同伴互评
- 作品或附件

评分维度：

- 规则意识
- 策略能力
- 合作能力
- 沟通能力
- 情绪稳定
- 创造性
- 复盘能力

输出：

- 游戏参与记录
- 游戏能力画像
- 游戏复盘报告
- 进入成长档案的关键事件

## 5.4 通识中心

通识中心负责记录学生如何理解世界、提出问题、形成判断。

- 通识主题
- 课程/讨论名称
- 老师
- 学生
- 参与时间
- 核心问题
- 关键概念
- 学生观点
- 讨论表现
- 阅读关联
- 作品关联

## 6. 外部平台导入与归档中心

这是系统长期可维护的关键模块。

### 6.1 支持来源

第一阶段支持手动或半自动导入：

- Google Sheet
- Excel
- CSV
- PDF
- Word
- 图片
- Google Drive 文件链接
- Notion 页面链接
- QUIP 文档链接
- Gmail 邮件导出
- Google Classroom 资料导出

后续再做自动接口：

- Google Drive API
- Gmail API
- Google Classroom API
- Notion API
- Google Sheets API

### 6.2 导入流程

不要让外部资料直接进入学生档案。  
必须先进“导入中心”。

流程：

1. 上传或连接外部资料
2. 系统识别学生姓名、时间、课程、老师、关键词
3. 尝试匹配学生学号
4. 匹配失败的进入“待确认”
5. 管理员或导师确认
6. 选择归档类型：动力记录、阅读记录、项目记录、作品、缴费、沟通等
7. 进入学生档案
8. 保留原始资料链接和导入记录

### 6.3 导入中心字段

- 来源平台
- 原始文件名/链接
- 上传人
- 上传时间
- 识别到的学生
- 匹配到的学号
- 置信度
- 归档类型
- 审核状态
- 审核人
- 审核时间
- 原始内容摘要
- 附件或链接

### 6.4 为什么不能直接搬运

过去的资料散落在多个平台，人工搬运成本高，也容易出现：

- 学生匹配错误
- 重复记录
- 资料丢失
- 时间错乱
- 权限混乱
- 无法追溯来源

所以必须采用“导入-匹配-确认-归档”的机制。

## 10. 权限与隐私

### 10.1 权限原则

- 最小可见
- 分角色授权
- 敏感内容分级
- 财务与成长记录隔离
- 家长端内容必须经过筛选
- 所有修改保留日志

### 10.2 可见性等级

每条记录建议设置可见性：

- 仅记录老师
- 导师可见
- 项目组可见
- 管理员可见
- 学生可见
- 家长可见

动力中心中较敏感的状态记录，不应默认同步给家长。

## 11. 维护与扩展口子

为了以后好维护、不容易出 bug，系统需要从第一版保留这些扩展口：

### 11.1 新增部门

通过新增：

- 部门类型
- 项目类型
- 权限组
- 记录模板

实现扩展，不改底层学生档案。

### 11.2 新增评价维度

评价维度必须可配置。  
例如未来新增“审美”“领导力”“国际理解”等，不需要重写代码。

### 11.3 新增外部平台

所有外部平台统一进入导入中心。  
新增平台只需要增加“来源适配器”，不影响学生档案主结构。

### 11.4 新增报告模板

报告模板可配置。  
未来可以有：

- 家长版
- 学生版
- 内部导师版
- 营期版
- 申请材料版
- 毕业档案版

### 11.5 保留审计日志

重要操作都要记录：

- 谁创建
- 谁修改
- 改了什么
- 什么时候改
- 是否导出
- 是否删除

## 12. 推荐技术方向

此处先给方向，不绑定具体开发团队。

### 12.1 产品形态

第一版建议：

- 管理后台：网页端
- 老师端：手机网页或小程序
- 学生/家长端：第二阶段再做完整小程序

### 12.2 后端能力

必须支持：

- 权限系统
- 文件上传
- 数据导入
- 搜索
- 报告生成
- 操作日志
- API 接口

### 12.3 数据库

建议使用关系型数据库作为主数据库。  
原因是学生、课程、项目、缴费、成绩、权限之间关系复杂，需要稳定、可查询、可追溯。

文件和附件使用对象存储，不要直接塞进数据库。

### 12.4 外部平台连接

系统需要预留 API 层：

- Google 系列
- Notion
- QUIP
- 邮件
- 企业微信/微信小程序

第一版先做导入，第二阶段再做自动同步。

## 15. 建议的下一步

下一步建议先做两件事：

1. 确认本蓝图中的 MVP 范围。
2. 设计老师端“动力中心快速记录页”的原型。
